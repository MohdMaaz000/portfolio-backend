Project 01

Book & Notes Management System

A full-stack study workspace where users register, log in, create books, and organise notes — accessible through both a browser UI and a REST API. Built to cover the complete backend surface: auth, migrations, validation, testing, and cloud deployment.

Session-based auth for the browser UI; JWT for the API layer. Alembic handles all schema migrations — the database is never modified by hand. SQLAlchemy models define the schema; Pydantic schemas handle request/response validation at the boundary.

Why Alembic instead of auto-migrate? SQLAlchemy's create_all() is fine for prototypes but gives you no migration history, no rollbacks, and no safe path to production. Alembic keeps every schema change versioned and reversible.

Why both a browser UI and a REST API? Server-rendered Jinja templates serve the UI without any frontend framework overhead, while the API layer keeps the backend independently testable and consumable by other clients. One codebase, two interfaces.

Task Manager API

Production-style REST API for task management. Flask, SQLAlchemy, and JWT authentication — with a full automated test suite. Built to demonstrate that a clean Flask API can match the structure and reliability expectations of a FastAPI project.

Why Flask and not FastAPI here? Flask is still the dominant choice in many production codebases. Understanding both frameworks — and their different approaches to routing, middleware, and testing — is more useful than knowing one deeply.

Why automated tests? The live API on Render is proof of deployment; the test suite is proof of engineering discipline. Tests cover auth edge cases (expired token, wrong credentials) and task state transitions, not just the happy path.

Project 03

MemoStack Backend

High-performance FastAPI backend for structured note-taking and collaborative book annotations. JWT access + refresh token auth, peer comment system, rate limiting, global request logging, and a clean services layer separating business logic from route handlers.

Client
→
FastAPI routerrate-limit · CORS · logging
→
Services layerbusiness logic
→
PostgreSQL / SQLiteprod · dev

Route handlers are thin — they validate input and call the service. The service layer owns all logic: token generation, ownership checks, comment eligibility. This keeps routes testable in isolation and business rules reusable across endpoints.

POST/auth/registerRegister, returns token pair
POST/auth/refreshRotate access token
GET/booksList user's books
POST/booksCreate book
POST/books/{id}/notesAdd note to book
PUT/notes/{id}Edit note
POST/notes/{id}/commentsPeer comment on note
DEL/comments/{id}Delete comment
POST /auth/register { "email": "peer@example.com", "password": "••••••••" } HTTP 201 Created { "access_token": "eyJhbGci...", "refresh_token": "eyJhbGci...", "token_type": "bearer" } ────────────────────────────────────────── POST /notes/12/comments Authorization: Bearer eyJhbGci... { "content": "Great summary of chapter 3." } HTTP 201 Created { "id": 44, "note_id": 12, "author": "peer@example.com", "content": "Great summary of chapter 3.", "created_at": "2025-04-08T11:40:00Z" }

Why access + refresh tokens? Short-lived access tokens (15 min) limit exposure if a token leaks. The refresh token rotates on use — if an attacker replays a stolen refresh token after it's been rotated, the server detects reuse and can revoke the session entirely.

Why a services layer? Without it, route handlers accumulate business logic until they're unreadable and untestable. The services layer enforces a boundary: routes handle HTTP, services handle logic. Swapping FastAPI for another framework later requires only changing the route layer.

Engineering approach

How I handle core backend concerns

authentication JWT access + refresh token pattern across all three projects. Access tokens are short-lived; refresh tokens rotate on use. In MemoStack, token reuse detection allows full session revocation. Auth logic lives in the service layer, not inside route handlers.
database design Schema managed through Alembic migrations — never create_all() in production. Foreign key relationships enforced at the DB level. SQLite for local dev, PostgreSQL for production; SQLAlchemy abstracts the difference without changing application code.
error handling Global exception handlers registered at the app level return a consistent error shape: { code, message, detail }. 4xx for client errors, 5xx for server failures — never reversed. Route handlers don't contain try/catch blocks.
validation Pydantic schemas define request and response shapes in FastAPI projects. Validation errors surface as structured 422 responses before any business logic runs. Flask project uses Marshmallow for the same boundary enforcement.
testing Pytest with a dedicated test database. Tests exercise the full stack — HTTP request through to DB query — not mocked services. Auth edge cases (expired tokens, wrong credentials, ownership violations) are tested explicitly, not assumed.
